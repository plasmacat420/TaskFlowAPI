Overview

What TaskFlow does

TaskFlow lets you create tasks, set their status and priority, assign them to team members, and track due dates. Nothing revolutionary on the surface — the point was never the features. The point was building something complete: a real API, a real database, a real deployed frontend, with all the rough edges that come with that.

It covers the full cycle — a browser sends an HTTP request, the .NET API processes it through several layers, EF Core writes or reads from PostgreSQL on Render, and the response comes back as JSON that Angular renders. Every step in that chain is real, not mocked.

Architecture

How the pieces fit together

The backend is split into four layers, each with a single responsibility. The rule is simple — a layer can only talk to the one directly below it. Controllers don't touch the database. Repositories don't contain business rules. Here's what that looks like in practice:

Why bother with four layers for a simple app? Because each layer can change independently. You could swap PostgreSQL for SQL Server by only changing the repository. You could add auth or caching to the service layer without touching the controllers. And in tests, you can swap the real repository for a fake one — the service doesn't know the difference because it only depends on the interface, not the implementation.
C# & .NET in Practice

The code, explained

Here are the key .NET patterns used throughout the backend — not as abstract concepts, but as they actually appear in this codebase.

Dependency Injection

Classes declare what they need, .NET provides it

Instead of a class creating its own dependencies with new, ASP.NET Core's built-in DI container injects them through the constructor. TaskService needs a repository — it just asks for ITaskRepository in its constructor and the framework handles the rest.

Everything database-related is registered as Scoped — one instance per HTTP request. This matters because TaskService, TaskRepository, and AppDbContext all share the same instance within one request, which means they share the same EF Core change tracker. Make DbContext a Singleton and two parallel requests would corrupt each other's data.

Program.cs
// Register as Scoped — shared within one HTTP request
builder.Services.AddScoped<ITaskRepository, TaskRepository>();
builder.Services.AddScoped<IUserRepository, UserRepository>();
builder.Services.AddScoped<ITaskService,    TaskService>();

// TaskService constructor — .NET injects both repositories automatically
public TaskService(ITaskRepository taskRepo, IUserRepository userRepo)
{
    _taskRepository = taskRepo;
    _userRepository = userRepo;
}
async / await

Never block a thread waiting for the database

Every method that touches the database is async. When a query is waiting on PostgreSQL, the thread is released back to the pool to handle other requests. A synchronous server blocks one thread per in-flight DB call — under load, you exhaust the thread pool fast. With async, a small number of threads can serve hundreds of concurrent requests.

The async Task<T> return type is C#'s way of saying "this method will eventually return a T, but not necessarily right now." await suspends the method until the work completes, then resumes — without blocking the thread.

TaskRepository.cs
public async Task<IEnumerable<TaskItem>> GetAllAsync()
{
    return await _context.Tasks
        .Include(t => t.AssignedUser)  // JOIN to users table
        .AsNoTracking()               // read-only — skip snapshot overhead
        .ToListAsync();               // executes SQL, releases thread while waiting
}
EF Core & LINQ

Write C# — EF Core writes the SQL

Entity Framework Core maps C# classes to database tables. The TaskItem class becomes the Tasks table; each property becomes a column. You never write raw SQL — you write LINQ, and EF Core translates it.

Important distinction: LINQ on IQueryable (a DbSet) builds an expression tree — no database call yet. The SQL only runs when you materialize it with ToListAsync(). After that, any further LINQ is in-memory C#. This is why .Where() before ToListAsync() becomes a SQL WHERE clause, but .Where() after it filters in memory.

TaskRepository.cs + TaskService.cs
// In the repository — runs as SQL WHERE clause
return await _context.Tasks
    .Where(t => t.AssignedUserId == userId)  // → SQL WHERE
    .Include(t => t.AssignedUser)             // → SQL JOIN
    .AsNoTracking()
    .ToListAsync();                           // ← SQL executes here

// In the service — runs in memory AFTER the DB call
return tasks.Select(TaskResponseDto.FromEntity);  // C# projection
DTOs

What the API sends is not what the database stores

The TaskItem entity has navigation properties like AssignedUser — the full User object loaded by EF Core. If you return that directly from the API, you'd serialize the entire user object nested in every task, risk circular reference exceptions, and couple your API contract to your database schema.

DTOs (Data Transfer Objects) are the translation layer. TaskResponseDto only carries what the frontend needs — AssignedUserName as a plain string, not the full User object. Rename a database column and nothing in the API response changes. The record keyword makes these immutable by default, which is exactly what a response shape should be.

TaskResponseDto.cs
public record TaskResponseDto
{
    public Guid           Id             { get; init; }
    public string         Title          { get; init; }
    public string?        AssignedUserName { get; init; }
    public TaskItemStatus Status         { get; init; }
    // entity has AssignedUser (full User object) — DTO flattens it to just the name

    public static TaskResponseDto FromEntity(TaskItem t) => new()
    {
        Id              = t.Id,
        Title           = t.Title,
        AssignedUserName = t.AssignedUser?.Name
    };
}
ASP.NET Core

Middleware — every request passes through a pipeline

Before any controller sees a request, it passes through a chain of middleware in Program.cs. Each piece can inspect the request, modify it, short-circuit with its own response, or pass it along. Order matters — a lot.

The middleware pipeline is also bidirectional: response flows back through the same chain in reverse. CORS headers must be added before the response is committed, so UseCors must be registered before anything that might write a response — including UseHttpsRedirection. This exact ordering caused a real bug in this app (more on that below).

Program.cs — middleware order
app.UseRouting();
app.UseCors("AllowFrontend");  // must be BEFORE https redirect
app.UseHttpsRedirection();
app.UseAuthorization();
app.MapControllers();
Frontend

The Angular side

The frontend is a standard Angular 17 app using NgModule (not standalone components) — the traditional enterprise pattern. Four components, two services, reactive forms, and RxJS for async data handling.

◈ Components

Dashboard shows task KPIs and a team overview. TaskList renders all tasks in a filterable card grid. TaskForm handles both create and edit in one component — it detects mode from the route parameter :id. UserList manages the team with inline editing.

⟳ Reactive Forms

Forms are built with FormGroup and FormControl rather than ngModel. The structure, defaults, and validation rules all live in the TypeScript class — not scattered in the HTML template. This makes them easier to test and extend. form.markAllAsTouched() triggers error display on submit.

≋ RxJS Patterns

forkJoin runs two HTTP calls in parallel and waits for both — used when loading a task alongside the user list for the edit form. takeUntil(destroy$) automatically cancels all subscriptions when the component is destroyed, preventing memory leaks from dangling HTTP observers.

⌾ Environment Files

environment.ts and environment.prod.ts hold the API base URL. At build time, ng build --configuration=production swaps the file automatically. The app code always imports from environment.ts — it never needs to know which environment it's running in.

task-form.component.ts — forkJoin + takeUntil
// Load task and users list in parallel — don't wait for one before starting the other
forkJoin({
  task:  this.taskService.getTaskById(this.taskId!),
  users: this.userService.getAllUsers()
})
.pipe(takeUntil(this.destroy$))  // cancel if component is destroyed mid-flight
.subscribe(({ task, users }) => {
  this.users = users;
  this.form.patchValue({ title: task.title, status: task.status });
});
Things that broke

Bugs worth remembering

These weren't hypothetical edge cases. They hit during actual testing against the live deployed app. Each one took real debugging to understand — which is why they're worth documenting.

01 CORS

CORS preflight blocked by HTTPS redirect

The Angular app on GitHub Pages was getting a "No Access-Control-Allow-Origin header" error. The CORS policy was configured correctly — but it wasn't running first. UseHttpsRedirection was registered before UseCors in the middleware pipeline.

When a browser sends an HTTP request to an HTTPS API, it first sends an OPTIONS preflight to check if cross-origin is allowed. That preflight hit the redirect middleware first and got a 301 response with no CORS headers. Browsers refuse to follow redirects for preflights — they just fail with a CORS error, which completely masks the real cause.

Fix: move UseCors above UseHttpsRedirection. Three lines reordered. Hours of head-scratching before that.
02 EF Core

Unassigning a task did nothing — silently

Setting a task's assignee to "None" would save with no error, but refresh the page and the original user was still there. The update was being silently ignored.

The repository uses .Include(t => t.AssignedUser) to eagerly load the user for the response DTO. When you then set task.AssignedUserId = null and call _context.Update(task), EF Core reconciles the foreign key value from the navigation property — not from the FK field. Since task.AssignedUser still held the User object from the original load, EF Core put the old ID straight back.

Fix: set task.AssignedUser = null first, then task.AssignedUserId = dto.AssignedUserId. Clearing the navigation property removes EF Core's "authoritative" source, so the FK value sticks.
03 Npgsql

DateTime crash — Kind=Unspecified rejected by PostgreSQL

Saving a task with a due date threw a DbUpdateException: "Cannot write DateTime with Kind=Unspecified to PostgreSQL type 'timestamp with time zone'". This only started in Npgsql 9, which became strict about it.

HTML's <input type="date"> gives back a string like "2025-06-01" — no timezone. System.Text.Json parses that as a DateTime with Kind=Unspecified. PostgreSQL's timestamptz column requires UTC, so Npgsql 9 rejects it outright.

Fix: Angular appends T00:00:00Z to the date string before sending. Backend also calls DateTime.SpecifyKind(dt, DateTimeKind.Utc) as a safety net. Both layers now guarantee UTC.
04 Render

Cold start disguised as a CORS error

After fixing the real CORS bug, CORS errors kept appearing on first load — but only sometimes, and only after the app had been idle for a while. The backend was fine; the CORS config was correct.

Render's free tier spins down containers after 15 minutes of inactivity. The first request after that hits a cold start and times out — no response at all. The browser interprets a missing response as missing CORS headers and reports a CORS error. The actual problem was a sleeping server, not cross-origin policy.

Fix: AppComponent.ngOnInit() now pings the API immediately on startup and shows a "Connecting..." banner. err.status === 0 detects a network timeout (vs an actual HTTP error) so the message is accurate.
